# Чек-лист доработок по итогам аудита

Ниже план работ по спринтам. Формат: `- [ ]` что сделать и почему это важно.

## Спринт 1 (P0: закрыть критичные риски)

- [x] Убрать все дефолтные пароли/секреты из `docker-compose*.yml` и шаблонов, оставить только пустые значения или явные `CHANGE_ME`. Почему: текущие дефолты позволяют компрометацию окружения.
- [ ] Ротация `SECRET_KEY`, паролей БД, RabbitMQ, MinIO, Grafana во всех окружениях. Почему: если ключи уже светились в истории/логах, ротация обязательна.
- [x] Закрыть внешние порты admin-сервисов (RabbitMQ UI, Grafana, Prometheus), оставить доступ только из приватной сети/VPN. Почему: открытые панели управления это прямой вектор атаки.
- [x] Ограничить/закрыть Swagger и OpenAPI в production (или вынести под admin auth/IP allowlist). Почему: упрощает разведку и подбор атак.
- [x] Включить строгий policy для prod старта: без безопасных секретов сервис не поднимается. Почему: исключает случайный запуск с небезопасной конфигурацией.

## Спринт 2 (P1: усилить аутентификацию и контроль доступа)

- [ ] Разделить ключи подписи JWT для user и superadmin токенов. Почему: компрометация одного ключа не должна давать полный доступ.
- [ ] Для superadmin добавить хешированный пароль, rate limit на login и lockout/backoff. Почему: снижает риск брутфорса.
- [ ] В JWT валидации проверять `type=access`, `iss`, `aud` и обязательные claims. Почему: предотвращает прием нецелевых/поддельных токенов.
- [ ] Устранить race condition в refresh-flow (`SELECT ... FOR UPDATE` или atomic update). Почему: исключает повторное использование refresh токена в гонке.
- [ ] Добавить аудит событий security-класса (failed login, forbidden, refresh anomalies). Почему: быстрее выявлять атаки и инциденты.

## Спринт 3 (P1/P2: устойчивость и безопасность данных)

- [ ] Перевести global rate limit на Redis-based (shared) limiter. Почему: in-memory limiter обходится при нескольких worker/process.
- [ ] Усилить лимит размера запроса для chunked upload и streaming. Почему: защита от DoS через большие тела без `Content-Length`.
- [ ] Для загрузки файлов внедрить потоковую обработку, лимиты размера, whitelist MIME и проверку содержимого. Почему: снижает риск OOM и вредоносных файлов.
- [ ] Для CSV/XLSX импорта и экспорта добавить защищенные лимиты (строки/колонки/время) и graceful-fail. Почему: предотвращает деградацию под нагрузкой.
- [ ] Убрать возврат деталей провайдера AI в клиентские ошибки, оставить безопасные коды. Почему: исключает утечки внутренних деталей интеграций.

## Спринт 4 (P2: фронтенд и инженерные практики)

- [ ] Уйти от хранения токенов в `localStorage` к `HttpOnly Secure SameSite` cookie-сессии. Почему: снижает риск кражи токена при XSS.
- [ ] Пересобрать CSP под фронтенд с nonce/hash и убрать лишние разрешения. Почему: уменьшает поверхность XSS.
- [ ] Сделать тесты и type-check блокирующими в CI (убрать `|| echo`). Почему: нельзя пропускать регрессии в production.
- [ ] Добавить security regression набор тестов (auth, token, ACL, tenant isolation). Почему: фиксирует критичные гарантии безопасности на будущее.
- [ ] Добавить регулярный dependency audit (pip/npm + CVE policy). Почему: снижает supply-chain риски.

## Definition of Done (для каждого спринта)

- [ ] Есть PR с чек-листом выполненных пунктов и ссылками на коммиты.
- [ ] Есть обновленный runbook по новым настройкам и ротации секретов.
- [ ] Есть подтверждение от QA/DevOps по smoke/regression после релиза.
