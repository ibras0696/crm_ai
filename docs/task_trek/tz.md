# Описание дизайна модуля «Таск-трекер» для CRM платформы

## 1. Назначение экрана

Экран предназначен для управления задачами внутри уже существующей SaaS CRM платформы.  
Модуль должен закрывать базовые сценарии:

- постановка задач;
- контроль выполнения;
- работа по Kanban-доске;
- контроль сроков и ответственных;
- просмотр деталей задачи;
- фиксация чек-листов, файлов, комментариев, зависимостей и истории изменений.

Дизайн сделан так, чтобы модуль выглядел частью текущей CRM, а не отдельным внешним сервисом.

---

## 2. Общая структура интерфейса

Экран делится на 4 основные зоны:

1. **Левое боковое меню**
2. **Верхняя панель аккаунта**
3. **Центральная рабочая область**
4. **Правая панель деталей выбранной задачи**

Схема расположения:

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│ Верхняя панель аккаунта / CRM                                               │
├───────────────┬───────────────────────────────────────────────┬─────────────┤
│               │                                               │             │
│ Левое меню    │ Центральная область таск-трекера              │ Правая      │
│ навигации     │ KPI + фильтры + Kanban-доска                  │ панель      │
│               │                                               │ задачи      │
│               │                                               │             │
└───────────────┴───────────────────────────────────────────────┴─────────────┘
```

## 3. Левое боковое меню

### Расположение

Левое меню находится по всей высоте экрана с левой стороны.

### Что содержит

В верхней части:

- логотип CRM;
- название продукта: **CRM Платформа**;
- подпись проекта / организации.

Далее идут разделы:

### Основное

- Главная
- Команда
- Журнал

### Модули

- Таблицы
- Чат
- Документы
- База знаний
- **Таск-трекер** — активный пункт
- Расписание
- Аналитика
- AI Агент
- Админ-панель
- Биллинг

### Нижняя часть меню

- Инструкции
- Настройки
- Свернуть

### Логика дизайна

Активный пункт **«Таск-трекер»** выделяется цветом.  
В тёмной теме — тёмно-синяя подсветка с ярким синим акцентом.  
В светлой теме — светло-голубой фон и синий текст.

---

## 4. Верхняя панель

### Расположение

Находится сверху, справа от логотипа и над рабочей областью.

### Что содержит

- переключатель темы;
- иконка уведомлений;
- бейдж тарифа **BUSINESS**;
- аватар пользователя;
- имя пользователя;
- email;
- иконка выхода / раскрытия меню.

### Назначение

Верхняя панель сохраняет общую структуру CRM и показывает, что таск-трекер встроен внутрь основного продукта.

---

## 5. Заголовок страницы

### Расположение

В верхней части центральной рабочей области.

### Содержание

```text
Таск-трекер
Планирование, отслеживание и контроль задач команды
```

В тёмной версии подзаголовок может быть:

```text
Планирование, исполнение и контроль задач команды
```

### Назначение

Пользователь сразу понимает, что он находится в модуле управления задачами, а не в базе знаний, документах или таблицах.

---

## 6. Панель действий и фильтров

### Расположение

Под заголовком страницы.

### Элементы

Слева направо:

1. **Создать задачу**
   - основная синяя кнопка;
   - должна открывать форму создания задачи.

2. **Поиск задач**
   - поле поиска по названию, ID, описанию, проекту, тегам.

3. **Фильтры**
   - общий фильтр по параметрам задачи.

4. **Ответственный / Исполнители**
   - фильтр по сотрудникам.

5. **Приоритет**
   - фильтр по уровню важности.

6. **Дедлайн**
   - фильтр по сроку выполнения.

7. **Переключатели вида**
   - Kanban;
   - список;
   - таблица;
   - календарь.

### Рекомендация для ТЗ

Минимально в первой версии нужны:

- создание задачи;
- поиск;
- фильтр по статусу;
- фильтр по ответственному;
- фильтр по приоритету;
- фильтр по дедлайну;
- Kanban-представление.

## 7. KPI-блок

### Расположение

Под панелью действий, над Kanban-доской.

### Карточки

| Карточка | Назначение |
|---|---|
| Просрочено | Показывает задачи, дедлайн которых уже прошёл |
| На сегодня | Показывает задачи с дедлайном сегодня |
| В работе | Показывает активные задачи |
| Завершено | Показывает выполненные задачи |

### Пример данных

```text
Просрочено: 7
На сегодня: 12
В работе: 24 / 28
Завершено: 36 / 56
```

### Визуальная логика

- Просрочено — красный акцент.
- На сегодня — оранжевый акцент.
- В работе — синий акцент.
- Завершено — зелёный акцент.

---

## 8. Kanban-доска

### Расположение

Основная центральная часть экрана.

### Колонки

1. **Бэклог**
2. **К выполнению**
3. **В работе**
4. **На проверке**
5. **Готово**

### Назначение колонок

| Колонка | Значение |
|---|---|
| Бэклог | Задачи, которые ещё не взяты в работу |
| К выполнению | Задачи, готовые к началу работы |
| В работе | Задачи, которые сейчас выполняются |
| На проверке | Задачи, ожидающие проверки / ревью |
| Готово | Завершённые задачи |

### Поведение

Задачи должны перемещаться между колонками drag-and-drop.  
При переносе задачи должен меняться её статус.

---

## 9. Карточка задачи

### Расположение

Карточки находятся внутри колонок Kanban-доски.

### Что показывает карточка

Каждая карточка задачи должна отображать минимум:

- название задачи;
- проект;
- приоритет;
- ответственного;
- дедлайн;
- чек-лист прогресс;
- количество комментариев;
- количество вложений;
- оценку времени;
- статус / визуальный цвет.

### Пример карточки

```text
Реализовать модуль отчетов
Проект: CRM Платформа

Приоритет: Высокий
Ответственный: ИГ
Дедлайн: 25 мая
Чек-лист: 3/7
Комментарии: 5
Время: 16ч
```

### Визуальные акценты

Приоритеты:

| Приоритет | Цвет |
|---|---|
| Низкий | зелёный |
| Средний | жёлтый / оранжевый |
| Высокий | красный / оранжевый |
| Критический | красный |

---

## 10. Правая панель деталей задачи

### Расположение

Правая панель открывается справа при клике на карточку задачи.

Она не должна уводить пользователя со страницы.  
Пользователь остаётся на доске и параллельно видит детали выбранной задачи.

### Верхняя часть панели

Содержит:

- статус задачи;
- ID задачи;
- кнопку закрытия;
- название задачи;
- короткое описание.

Пример:

```text
В работе · ID: TASK-1247

Разработать модуль уведомлений

Создать универсальный модуль уведомлений для внутрисистемных и email-уведомлений.
```

1. **Детали**

### Назначение

| Вкладка | Что внутри |

| Комментарии | Обсуждение задачи |

## 12. Поля задачи для ТЗ

Ниже структура данных, которую нужно заложить в систему.

### Основные поля

| Поле | Тип | Обязательное | Описание |
|---|---:|---:|---|
| ID задачи | string / UUID | Да | Уникальный идентификатор задачи |
| Название | string | Да | Короткое название задачи |
| Описание | text | Нет | Подробное описание задачи |
| Проект | relation | Да | Проект, к которому относится задача |
| Этап / Статус | enum | Да | Текущий статус задачи |
| Ответственный | user relation | Да | Исполнитель задачи |
| Постановщик | user relation | Да | Кто поставил задачу |
| Приоритет | enum | Да | Низкий, средний, высокий, критический |
| Дедлайн | datetime | Нет | Срок выполнения |
| Дата начала | datetime | Нет | Когда задачу начали выполнять |
| Дата завершения | datetime | Нет | Когда задачу закрыли |

### Поля времени

| Поле | Тип | Описание |
|---|---:|---|
| Оценка времени | number | Плановое время в часах |
| Фактическое время | number | Реально затраченное время |
| Таймер | boolean / timer entity | Возможность учитывать время работы |

### Командные поля

| Поле | Тип | Описание |
|---|---:|---|
| Наблюдатели | user[] | Люди, которые следят за задачей |
| Участники | user[] | Дополнительные исполнители |
| Комментарии | comment[] | Обсуждение задачи |
| Упоминания | mention[] | Уведомления через @username |

### Организационные поля

| Поле | Тип | Описание |
|---|---:|---|
| Теги | tag[] | Метки задачи |
| Чек-лист | checklist[] | Список пунктов выполнения |
| Подзадачи | task[] | Дочерние задачи |
| Зависимости | task[] | Задачи, без которых нельзя завершить текущую |
| Связанные задачи | task[] | Логически связанные задачи |
| Файлы / вложения | file[] | Документы, скриншоты, изображения |
| SLA / срок реакции | datetime / duration | Время, за которое нужно отреагировать |
| Напоминания | reminder[] | Уведомления по срокам |
| История изменений | audit_log[] | Кто, что и когда изменил |

## 16. Ключевые UX-удобства

### 1. Быстрое создание задачи

Кнопка **Создать задачу** должна открывать форму или модальное окно.

Минимальные поля:

- название;
- проект;
- ответственный;
- статус;
- приоритет;
- дедлайн;
- описание.

### 2. Просмотр без ухода со страницы

Правая панель позволяет открыть задачу и не терять контекст доски.

### 3. Drag-and-drop

Перемещение карточек между колонками должно менять статус.

### 4. Быстрые фильтры

Пользователь должен быстро найти:

- свои задачи;
- просроченные задачи;
- задачи на сегодня;
- задачи по проекту;
- задачи по ответственному;
- задачи по приоритету.

### 5. Контроль сроков

Просроченные задачи должны быть визуально заметны.

### 6. История изменений

Важна для корпоративной CRM, чтобы видеть:

- кто изменил статус;
- кто поменял дедлайн;
- кто назначил ответственного;
- кто добавил комментарий или файл.

## 19. Главная идея дизайна

Дизайн должен быть не просто красивым, а рабочим для реального SaaS-продукта.

Главный принцип:

> Пользователь должен за 5 секунд понять, какие задачи горят, кто за них отвечает, что сейчас в работе и где есть задержки.

Поэтому на экране одновременно видны:

- общая картина через KPI;
- рабочий процесс через Kanban;
- детали конкретной задачи через правую панель;
- фильтры и поиск для быстрого управления.
