# ТЗ-описание двух экранов таск-трекера CRM

Документ описывает два важных раздела модуля **«Таск-трекер»**:

1. **Экран конкретной задачи**
2. **Экран создания задачи**

Эти разделы должны быть встроены в существующий продакшн SaaS CRM-проект и визуально совпадать с текущей системой: левое меню, верхняя панель, светлая и тёмная темы, единая сетка, единые кнопки, бейджи и состояния.

---

# 1. Экран конкретной задачи

## 1.1. Назначение экрана

Экран конкретной задачи нужен для полноценной работы с одной задачей:

- просмотреть всю информацию по задаче;
- понять статус, дедлайн, ответственного и приоритет;
- проверить чек-лист;
- посмотреть подзадачи;
- прочитать комментарии;
- открыть файлы;
- увидеть активность и историю изменений;
- отредактировать задачу;
- закрыть или перевести задачу в другой статус.

Главная идея экрана:

> Пользователь должен открыть задачу и сразу понять: что нужно сделать, кто отвечает, насколько задача срочная, что уже выполнено и что блокирует завершение.

---

## 1.2. Общая структура экрана

Экран состоит из 4 основных зон:

1. **Левое меню CRM**
2. **Верхняя панель аккаунта**
3. **Основная область задачи**
4. **Правая информационная панель**

Схема:

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Верхняя панель CRM                                                           │
├──────────────┬──────────────────────────────────────────────┬────────────────┤
│              │                                              │                │
│ Левое меню   │ Основная область задачи                      │ Правая панель  │
│ CRM          │ Описание, чек-лист, подзадачи, комментарии   │ Детали задачи  │
│              │                                              │ Файлы, история │
│              │                                              │                │
└──────────────┴──────────────────────────────────────────────┴────────────────┘
```

---

## 1.3. Левое меню

Левое меню остаётся таким же, как в общей CRM.

Активный пункт:

```text
Таск-трекер
```

Он должен быть выделен цветом.

### Требование

Экран конкретной задачи не должен выглядеть как отдельная страница вне CRM.  
Пользователь должен понимать, что он всё ещё находится внутри модуля **«Таск-трекер»**.

---

## 1.4. Верхняя навигация задачи

В верхней части основной области нужно показать хлебные крошки:

```text
Таск-трекер / Sprint 12 / TASK-1247
```

### Назначение

Хлебные крошки нужны, чтобы пользователь понимал:

- из какого раздела он пришёл;
- к какому спринту / доске относится задача;
- какой ID у задачи.

### Поведение

- клик по **Таск-трекер** возвращает на Kanban-доску;
- клик по **Sprint 12** открывает текущую доску / спринт;
- ID задачи просто отображается как текущий объект.

---

## 1.5. Заголовок задачи

Под хлебными крошками находится главный заголовок:

```text
Разработать модуль уведомлений
```

Рядом можно добавить иконку избранного / закрепления:

```text
☆
```

### Под заголовком отображаются быстрые параметры:

```text
В работе · Высокий · CRM Платформа · Дедлайн: 22 мая · Оценка: 16ч · Факт: 8ч
```

### Обязательные элементы

| Элемент | Назначение |
|---|---|
| Статус | показывает текущий этап задачи |
| Приоритет | показывает срочность |
| Проект | связь с проектом |
| Дедлайн | срок выполнения |
| Оценка времени | плановое время |
| Фактическое время | уже затраченное время |

---

## 1.6. Верхние действия

Справа от заголовка или над правой панелью должны быть кнопки:

```text
Редактировать
Поделиться
История
Ещё
```

### Кнопка «Редактировать»

Открывает режим редактирования задачи или отдельную форму.

### Кнопка «Поделиться»

Копирует ссылку на задачу.

### Кнопка «История»

Открывает историю изменений.

### Кнопка «Ещё»

Выпадающее меню:

- дублировать задачу;
- переместить в другой проект;
- архивировать;
- удалить;
- отметить как заблокированную;
- создать подзадачу;
- создать связанную задачу.

---

## 1.7. Вкладки задачи

Под заголовком должны быть вкладки:

```text
Обзор | Комментарии | Подзадачи | Файлы | Активность
```

### Назначение вкладок

| Вкладка | Что показывает |
|---|---|
| Обзор | основное описание, чек-лист, прогресс, связанные задачи |
| Комментарии | обсуждение задачи |
| Подзадачи | дочерние задачи |
| Файлы | вложения |
| Активность | история действий и изменений |

### Рекомендация

По умолчанию открывать вкладку **Обзор**.

# 2. Основная область экрана задачи

## 2.1. Блок «Описание»

### Расположение

В верхней левой части основной области.

### Содержание

```text
Описание

Создать универсальный модуль уведомлений для внутренних системных и email-уведомлений.
Модуль должен поддерживать шаблоны, каналы доставки, журнал событий, email и гибкие правила отправки.

Включает:
- настройки предпочтений пользователя;
- журнал уведомлений;
- API для других модулей;
- email-уведомления;
- системные уведомления внутри CRM.
```

### Требования

Описание должно поддерживать:

- обычный текст;
- списки;
- ссылки;
- выделение жирным;
- markdown или rich text;
- упоминания пользователей через `@`;
- ссылки на другие задачи через `TASK-1247`.

---

## 2.2. Блок «Прогресс»

### Расположение

Справа от описания или в верхней части второй колонки основной области.

### Содержание

```text
Прогресс
60%
4 / 7 выполнено
```

### Визуализация

- процент выполнения;
- progress bar;
- количество выполненных пунктов чек-листа.

### Логика расчёта

Прогресс может считаться по чек-листу:

```text
Выполненные пункты / Все пункты * 100
```

Пример:

```text
4 / 7 = 57%, округляется до 60%
```

---

## 2.3. Блок «Чек-лист»

### Расположение

Под описанием.

### Содержание

```text
Чек-лист 4/7

☑ Проанализировать структуру данных
☑ API для получения отчётов
☑ Реализация бизнес-логики
☑ UI: таблицы и фильтры
☐ Интеграция с email-сервисом
☐ Юнит-тесты
☐ Документация и примеры использования
```

### Функции

- добавить пункт;
- отметить пункт выполненным;
- снять выполнение;
- удалить пункт;
- изменить название пункта;
- перетащить пункт выше / ниже.

### UX-требование

Пользователь должен иметь возможность быстро отмечать выполнение без открытия отдельного окна.

---

## 2.4. Блок «Подзадачи»

### Расположение

Рядом с прогрессом или ниже него.

### Содержание

```text
Подзадачи 4

T-1247-1  Проектирование моделей данных      Готово
T-1247-2  Реализация API уведомлений         В работе
T-1247-3  Шаблоны UI и настройки             В работе
T-1247-4  Интеграция с email-сервисом        Открыто
```

### Что показывать у подзадачи

| Поле | Описание |
|---|---|
| ID | короткий номер подзадачи |
| Название | что нужно сделать |
| Статус | текущий этап |
| Ответственный | аватар или инициалы |
| Дедлайн | если есть |

### Поведение

- клик по подзадаче открывает её как отдельную задачу;
- можно добавить новую подзадачу;
- можно быстро изменить статус.

---

## 2.5. Блок «Связанные задачи / зависимости»

### Расположение

Под чек-листом.

### Содержание

```text
Связанные задачи / зависимости

TASK-145  Доработка фильтров              На проверке
TASK-150  Интеграция с модулем пользователей  Готово
TASK-148  Обновление базы данных          В работе
```

### Типы связей

| Тип связи | Значение |
|---|---|
| Зависит от | текущую задачу нельзя завершить без другой задачи |
| Блокирует | текущая задача блокирует другую |
| Связана с | логическая связь без жёсткой зависимости |
| Дубликат | похожая или повторяющаяся задача |

### Требование

Если есть блокирующая зависимость, это должно быть визуально заметно.

---

## 2.6. Блок «Комментарии»

### Расположение

В нижней части основной области или во вкладке **Комментарии**.

### Содержание

```text
Комментарии 5

Алексей Смирнов · 23 мая 11:40
Схема БД готова, отправлю на ревью.

Ибрагим Гатаев · 23 мая 12:15
Отлично. После ревью сразу отпишись по замечаниям.
```

### Функции

- написать комментарий;
- ответить на комментарий;
- упомянуть пользователя;
- прикрепить файл;
- редактировать свой комментарий;
- удалить свой комментарий;
- закрепить важный комментарий.

### UX-требование

Поле комментария должно быть доступно без лишних кликов.

# 4. Экран создания задачи

## 4.1. Назначение экрана

Экран создания задачи нужен, чтобы быстро и полноценно поставить задачу сотруднику или команде.

Главный принцип:

> Создание задачи должно быть быстрым для простых задач и достаточно подробным для сложных задач.

То есть пользователь должен иметь возможность создать задачу за 20–30 секунд, но при необходимости заполнить расширенные поля.

---

## 4.2. Общая структура экрана создания

Экран делится на 4 зоны:

1. **Левое меню CRM**
2. **Верхняя панель**
3. **Основная форма**
4. **Правая вспомогательная колонка**

Схема:

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Верхняя панель CRM                                                           │
├──────────────┬──────────────────────────────────────────────┬────────────────┤
│              │                                              │                │
│ Левое меню   │ Основная форма создания задачи               │ Правая колонка │
│ CRM          │ Название, описание, чек-лист, файлы           │ Детали,        │
│              │                                              │ шаблоны,       │
│              │                                              │ подсказки      │
└──────────────┴──────────────────────────────────────────────┴────────────────┘
```

---

## 4.3. Верхняя часть формы

### Хлебные крошки

```text
Таск-трекер / Создание задачи
```

### Заголовок

```text
Новая задача
```

### Подзаголовок

```text
Создайте задачу, назначьте исполнителей и спланируйте работу.
```

### Верхние кнопки

Справа сверху:

```text
Сохранить как черновик
Отмена
Создать задачу
```

### Автосохранение

Рядом можно показывать статус:

```text
Черновик сохраняется автоматически
```

# 5. Основная форма создания задачи

## 5.1. Поле «Название задачи»

### Расположение

В самом верху основной формы.

### Тип

```text
input text
```

### Обязательное

Да.

### Placeholder

```text
Например: Реализовать модуль отчётов для клиентов
```

### Валидация

- минимум 3 символа;
- максимум 150 символов;
- нельзя создать задачу без названия.

---

## 5.2. Поле «Краткое описание»

### Тип

```text
input text / textarea small
```

### Назначение

Коротко объяснить суть задачи в 1–2 строки.

### Пример

```text
Модуль должен позволять формировать кастомные отчёты и экспортировать их в PDF и Excel.
```

---

## 5.3. Поле «Описание»

### Тип

```text
rich text editor
```

### Возможности

- обычный текст;
- заголовки H1 / H2;
- списки;
- чекбоксы;
- ссылки;
- упоминания пользователей;
- вставка кода;
- вставка изображений;
- markdown shortcuts.

### Пример

```text
Необходимо разработать модуль отчётов, который позволит клиентам:

- создавать кастомные отчёты с гибкими фильтрами;
- экспортировать данные в PDF и Excel;
- сохранять шаблоны отчётов;
- ограничивать доступы по ролям.
```

---

## 5.4. Быстрые вкладки под описанием

Под редактором можно добавить быстрые действия:

```text
Чек-лист | Подзадача | Файл | Упоминание | Шаблон
```

### Назначение

Эти кнопки ускоряют заполнение задачи.

---

## 5.5. Чек-лист при создании

### Расположение

Под описанием.

### Содержание

```text
Чек-лист

☐ Проанализировать требования
```

### Требования

- можно добавлять пункты до создания задачи;
- можно удалять пункты;
- можно менять порядок;
- чек-лист сохраняется вместе с задачей.

---

## 5.6. Подзадачи при создании

### Расположение

Рядом с чек-листом или ниже него.

### Содержание


При создании главной задачи система должна создать связанные дочерние задачи.

---

## 5.7. Вложения

### Расположение

В нижней части основной формы.

### Содержание

```text
Вложения
```

### Требования

- drag-and-drop загрузка;

- отображение прогресса загрузки;

- ограничение по размеру и типу файла.

## 6.2. Обязательные поля создания

| Поле | Обязательное | Причина |
|---|---:|---|
| Название | Да | Без названия задача бессмысленна |
| Проект | Да | Задача должна принадлежать проекту |
| Статус | Да | Нужно понимать этап |
| Исполнитель | Желательно / Да по настройке | Для управляемости |
| Постановщик | Да | Для ответственности |
| Приоритет | Да | Для сортировки и контроля |
| Дедлайн | Нет / Да по настройке | Зависит от политики компании |

### Рекомендация

Для CRM лучше сделать настройку на уровне компании:

```text
Можно ли создавать задачу без исполнителя?
Можно ли создавать задачу без дедлайна?
Можно ли создавать задачу без проекта?
```

# 7. Логика кнопок создания

## 7.1. Кнопка «Создать задачу»

### Поведение

При клике:

1. валидируются обязательные поля;
2. создаётся задача;
3. создаются чек-листы;
4. создаются подзадачи;
5. загружаются вложения;
6. создаётся запись в истории;
7. отправляются уведомления ответственным;
8. пользователь переходит на экран созданной задачи или возвращается на доску.

### Рекомендация

После создания показывать toast:

```text
Задача создана
Открыть задачу | Вернуться на доску
```

---

## 7.2. Кнопка «Сохранить как черновик»

### Назначение

Сохраняет задачу без публикации и уведомлений.

### Черновик должен хранить:

- название;
- описание;
- заполненные поля;
- чек-лист;
- подзадачи;
- вложения;
- теги.

---

## 7.3. Кнопка «Отмена»

### Поведение

Если форма пустая — сразу возвращает назад.

Если есть введённые данные — показать подтверждение:

```text
Вы уверены, что хотите выйти?
Несохранённые данные будут потеряны.

Остаться
Выйти без сохранения
Сохранить черновик
```

```http
GET /api/tasks/{task_id}
```

Возвращает:

- основную задачу;
- чек-лист;
- подзадачи;
- комментарии;
- вложения;
- связанные задачи;
- активность;

# 10. Важные UX-состояния

## 10.1. Loading

При открытии задачи:

```text
Загружаем задачу...
```

Показывать skeleton-блоки вместо пустого экрана.

## 10.2. Empty states

Если нет комментариев:

```text
Комментариев пока нет. Напишите первый комментарий.
```

Если нет файлов:

```text
Файлы ещё не добавлены.
```

Если нет подзадач:

```text
Подзадач пока нет. Добавьте первую подзадачу.
```

## 10.3. Error state

Если задача не найдена:

```text
Задача не найдена или у вас нет доступа.
Вернуться к доске
```

## 10.4. Unsaved changes

При редактировании или создании задачи:

```text
У вас есть несохранённые изменения.
Сохранить черновик | Выйти без сохранения | Остаться
```

# 11. Итоговая рекомендация

Для продакшн CRM лучше делать не просто форму и страницу, а полноценную рабочую систему вокруг задачи.

Минимум для качественного MVP:

## Экран задачи

- заголовок;
- статус;
- приоритет;
- исполнитель;
- дедлайн;
- описание;
- чек-лист;
- комментарии;
- подзадачи;
- файлы;
- активность;
- правая панель деталей.

## Экран создания задачи

- название;
- описание;
- проект;
- статус;
- приоритет;
- исполнитель;
- дедлайн;
- чек-лист;
- файлы;
- теги;
- шаблоны;
- сохранение черновика.

Главное правило:

> Задача должна быть не просто записью в таблице, а рабочим объектом, вокруг которого команда ведёт исполнение, обсуждение, контроль и историю.
