# Анализ проекта CRM_AI

Проект представляет собой мультитенантную CRM / SaaS-платформу со встроенным Airtable-подобным конструктором таблиц, модулем базы знаний, календарем, порталом отчетов и AI-агентом.

## 🏗 Резюме архитектуры

Платформа состоит из:
1. **Бэкенд:** FastAPI (Python 3.12+), SQLAlchemy (Асинхронный), PostgreSQL + pgvector (для AI/RAG), Redis (кэш, Celery broker), Celery (фоновые задачи).
2. **Фронтенд:** React, Vite, TypeScript, TailwindCSS. Использует Radix UI для примитивов компонентов, `zod` и `react-hook-form` для форм, `react-query` для стейт-менеджмента API, а также тяжелые библиотеки (`ckeditor5`, `monaco-editor`, `pdf-lib`, `yjs` для возможной реализации совместного редактирования).
3. **Инфраструктура:** Docker, docker-compose (dev/prod варианты), Nginx, Prometheus + Grafana для мониторинга. RabbitMQ также заявлен в архитектуре (возможно, для Celery, судя по README).

## 🧩 Анализ модулей (Бэкенд)

Слой `backend/src/modules` имеет модульный монолитный дизайн и включает 15+ директорий, среди которых основные:
- **`auth`, `org`, `access`**: Отвечают за мультитенантность, аутентификацию (наверняка JWT), ролевую модель (RBAC/Rules) и приглашения.
- **`tables`**: Движок динамических таблиц (аналог Airtable). Данные таблиц (строки) хранятся в общей PostgreSQL таблице `table_records` в колонке типа `JSONB`. Имеется поддержка типов полей, представлений (Views) и папок.
- **`ai`**: Встроенный ИИ. Присутствует управление промптами, историей чатов (chat_controller), лимитами использования (limits.py) и логика RAG. 
- **`billing`, `superadmin`**: Внутренняя структура для управления тарифами (SaaS) и админ-панель. 
- **`knowledge`**: База знаний (типа Notion), страницы сохраняют структуру. На фронтенде видимо используется CKEditor.

## ⚙️ Анализ Фронтенда

Фронтенд организован в `Feature-Sliced` или схожей модульной структуре внутри `pages/` (папки админки, аналитики, таблиц, базы знаний и т.д.).
- Состояние управляется через `react-query`.
- Компоненты UI реализованы через `shadcn/ui` (опираясь на Radix) или схожий паттерн.
- Роутинг на базе `react-router-dom`.
Код выглядит достаточно современным, используются жесткие проверки типов (TS, Zod).

## ⚠️ Выявленные слабые места (Weak points)

1. **Производительность движка таблиц (JSONB):**
   * Все данные пользовательских динамических колонок сохраняются в один `JSONB` столбец. При росте объема данных в таблице (миллионы строк) сортировка и фильтрация по вложенным ключам JSONB без специализированных (GIN) индексов станет критическим узким местом.
   * `normalize_positions` в `records.py` для пересчета порядка строк (Drag & Drop) использует загрузку всех записей таблицы в память и цикл с обновлением позиций $O(N)$. Это приведет к дэдлокам или огромным задержкам на больших таблицах.
2. **Низкое покрытие тестами:** 
   * В директории `backend/tests` обнаружены только точечные тесты (`test_health.py`, `test_security_regressions.py`). Также в модуле `tables` есть всего 1 файл `test_tables_api.py`.
   * Отсутствуют (или их крайне мало) unit-тесты для сложной бизнес-логики (AI, права доступа, биллинг) и E2E тесты на фронтенде.
3. **Высокая монолитность:** 
   * 15 модулей в одном FastAPI сервисе. При росте команды это приведет к конфликтам при слияниях (Merge Conflicts) и сильной зацепленности кода (Coupling), если нет строгих правил межмодульного взаимодействия (Enforced Boundaries).
4. **Тяжелый бандл Frontend'a:** 
   * Наличие `monaco-editor`, `ckeditor5`, `pdfjs`, `fabric`, `mammoth` в одном проекте требует очень грамотного Code Splitting. В противном случае `First Load JS` будет весить мегабайты, что замедлит загрузку сайта для пользователей с медленным интернетом.
5. **Безопасность AI (Prompt Injection):** 
   * Если табличные данные, загруженные пользователем, попадают прямо в контекст LLM, существует риск извлечения данных других организаций (при слабой фильтрации RAG) через промпт-инъекции. Зависит от реализации `access_control` в `ai` модуле.

## 💡 Предложения по улучшению (Как бы я сделал этот проект лучше)

### 1. Архитектурные улучшения движка таблиц
- **Индексы для JSONB**: Добавить механизм создания GIN/BTREE индексов для часто-фильтруемых или сортируемых пользователем колонок.
- **Разреженный порядок (Sparse Ordering)**: Заменить $O(N)$ пересчет `position` (где счетчик 1, 2, 3...) на лексикографические индексы (например, string-based sorting вроде `LexoRank` от Jira) или использование рациональных чисел для `position`. Это позволит вставлять строку между двумя другими за $O(1)$ без обновления остальных записей в БД.

### 2. Качество кода и тестирование
- **Внедрить пайплайны CI/CD**: Добавить GitHub Actions или GitLab CI:
   - Линтеры (уже есть `ruff` для Python и `eslint` для TypeScript).
   - Запуск `pytest` с проверкой покрытия (Coverage). Отказ в мердже при низком % покрытия.
- **Добавить E2E тестирование**: Интегрировать `Playwright` или `Cypress` для тестирования критических путей (авторизация, создание таблиц, взаимодействие с AI базой знаний).
- **Разделение слоев (Hexagonal/Clean Architecture)**: В бэкенде более строго разделить контроллеры (FastAPI routes) от доменной логики (сервисы), чтобы сервисы могли тестироваться вообще без поднятия HTTP контекста (mocking).

### 3. Оптимизация Frontend'a
- Убедиться, что тяжелые редакторы (CKEditor, Monaco) грузятся асинхронно через `React.lazy()` или динамический `import()` инструмента Vite. 
- Настроить `rollupOptions.output.manualChunks` в `vite.config.ts`, чтобы библиотеки выносились в отдельные `.js` чанки и кешировались браузером.

### 4. Оптимизация AI
- **Семантическое разделение tenant'ов в pgvector**: Обособить вектора разных организаций (Tenant-wise partitions) или использовать RLS (Row-Level Security) в PostgreSQL, чтобы исключить риск попадания контекста компании А в чат компании Б.
- **Асинхронные стримы (Streaming Response)**: Убедиться, что ответы от AI приходят через Server-Sent Events (SSE) в FastAPI и отображаются по мере генерации на клиенте для лучшего UX. (Возможно, это уже сделано, но стоит проверить).

### 5. Инфраструктура
- Внедрить **Redis Cluster** для масштабирования Celery брокера и кэша, когда проект вырастет.
- Использовать **Terraform** для развертывания prod-инфраструктуры вместо ручного выполнения `docker-compose.prod.yml` на ВПС, если планируется SLA SaaS-уровня.
